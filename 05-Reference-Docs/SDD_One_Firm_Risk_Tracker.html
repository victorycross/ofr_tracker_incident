0. About This Document

0.1 Purpose

This Security Design Document (SDD) describes the security architecture, controls, and operational requirements for the One Firm Risk Tracker (OFR-RT) application. The OFR-RT is a single-page web application deployed as an Azure Static Web App that provides structured risk and issue tracking for PwC Canada's One Firm Risk group, replacing a manually maintained Word-based register with a modern, SharePoint-connected dashboard.

This document is authored in accordance with the PwC Security Design Document for Applications v1.5 template and is intended to support the Application Readiness Review (ARR) process, including completion of the Application Compliance Questionnaire (ACQ).

1. Solution Description

1.1 Application Description

The One Firm Risk Tracker (OFR-RT) is a client-side React single-page application (SPA) that provides:

  • Risk Intake Panel — structured submission of new risks/issues with owner, priority, and description
  • Issue Lifecycle Tracker — filterable, sortable tracker with staleness indicators (days since last update)
  • Update Workflow — modal-driven status updates with full audit history
  • Weekly Review Dashboard — KPI cards for total items, high-priority count, overdue items, and recently updated
  • CSV Export — denormalized export with UTF-8 BOM for SharePoint List import
  • Bilingual Support — full English / French (EN/FR) toggle
  • Dual-Mode Operation — Demo mode (in-memory sample data) and Live mode (SharePoint via Microsoft Graph API)

The application uses Microsoft Entra ID (Azure AD) for corporate SSO authentication and the Microsoft Graph REST API v1.0 to perform CRUD operations against three SharePoint Online Lists used as the data store.

CI Reference: Application to be registered in Archer upon completion of ARR. Configuration Items will be created for both the Azure Static Web App resource and the Entra ID App Registration.

3. Application Security Controls

3.1 Authentication & Access Methods

The application uses Microsoft Entra ID as the sole identity provider, implementing the OAuth 2.0 Authorization Code flow with PKCE for single-page application authentication.

Authentication Configuration

  • Protocol: OAuth 2.0 / OpenID Connect
  • Flow: Authorization Code with PKCE (Proof Key for Code Exchange)
  • Library: MSAL.js v2 (@azure/msal-browser 5.2.0)
  • Authority: https://login.microsoftonline.com/{tenant_id}
  • Token Cache: sessionStorage (cleared on tab close)
  • Cookie Usage: None (storeAuthStateInCookie: false)
  • MFA: Enforced via Entra ID Conditional Access policies (tenant-level)
  • PII Logging: Disabled in MSAL logger callback

Authentication Methods by Account Type

Account TypeAuth MethodMFAScopeSession Duration
Internal User (PwC Employee)Entra ID SSO via MSAL redirectYes (Conditional Access)User.Read, Sites.ReadWrite.AllTab session (sessionStorage)
External UserN/A — Not SupportedN/AN/AN/A
Privileged AdminSame as Internal + Entra PIMYes (enforced)Same + SharePoint Site AdminTab session
Service AccountN/A — No service accounts; delegated onlyN/AN/AN/A
Key Design Decision: The application uses exclusively delegated permissions (never application-level). All Graph API calls execute in the signed-in user's context, meaning access is governed by the user's SharePoint permissions plus the app's delegated scopes. There are no service accounts, background processes, or daemon flows.

3.4 Firewall Requirements & Application Segmentation

The OFR-RT is a client-side only application hosted on Azure Static Web Apps. There are no backend servers, VMs, containers, or Kubernetes clusters. All communication occurs from the browser to Microsoft-managed endpoints over HTTPS.

Application Segmentation

  • Browser Sandbox: All code executes within the browser's JavaScript sandbox with Same-Origin Policy protections
  • Azure Static Web App: Serverless; no VNet integration required
  • SharePoint Online: Data within M365 tenant boundary, governed by M365 Compliance policies

3.7 Security Stack

1. Access to Application

  • Microsoft Entra ID: Sole identity provider; single-tenant restricts to PwC Canada members only
  • Conditional Access: Tenant-level MFA, device compliance, location restrictions
  • PKCE Flow: Prevents authorization code interception attacks

2. Network Security

  • HTTPS-Only: Azure SWA enforces HTTPS with automatic redirect
  • TLS 1.2+: Minimum version enforced by all Microsoft endpoints
  • Same-Origin Policy: Browser-enforced; CORS not required
  • Content Security Policy: Configurable via staticwebapp.config.json

3. Data Security

  • Encryption at Rest: AES-256 via Microsoft platform-managed keys
  • Encryption in Transit: TLS 1.2+ for all communications
  • No Local Persistence: All data in M365 cloud; nothing stored locally
  • Token Isolation: sessionStorage per-tab isolation; cleared on close

4. Application Security

  • XSS Prevention: React auto-escapes all rendered content; no raw HTML injection patterns used
  • Dependency Scanning: GitHub Dependabot for vulnerability alerts
  • Static Analysis: ESLint with React hooks rules
  • Tree-Shaking: Vite removes unused code from production bundles

5. Container / Cluster Security

Not Applicable: The OFR-RT is deployed as an Azure Static Web App — a fully serverless, CDN-based hosting service. There are no containers, Kubernetes clusters, VMs, or server-side compute. This section is included for SDD template completeness.

3.8 Software Development Life Cycle (SDLC)

Source Code Management

  • Repository: GitHub (PwC Enterprise organization)
  • Branch Strategy: Git Flow — main (production), develop (integration), feature branches
  • Code Review: All merges to main require 1+ approving review via Pull Request
  • Branch Protection: main protected; no direct pushes; status checks required

CI/CD Pipeline

StageToolActions

Deploy (Staging)Azure SWA ActionAuto-deploy on PR; creates preview URL
Deploy (Production)Azure SWA ActionAuto-deploy on merge to main
RollbackGit revert + re-deployRevert commit triggers automatic re-deploy

Environment Configuration

Environment-specific values injected via Vite environment variables (VITE_ prefix), compiled into the static bundle at build time:

  • VITE_CLIENT_ID — Entra ID Application (Client) ID
  • VITE_TENANT_ID — Entra ID Directory (Tenant) ID
  • VITE_SP_HOSTNAME — SharePoint tenant hostname
  • VITE_SP_SITE_PATH — SharePoint site path
No Secrets in Code: The OAuth 2.0 public client (PKCE) flow does not require a client secret. Client ID and Tenant ID are non-secret, publicly embeddable identifiers. Sensitive operations are protected by the user's authentication token.
