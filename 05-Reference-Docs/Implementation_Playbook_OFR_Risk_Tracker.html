1. Playbook Overview

What Are We Deploying?

The One Firm Risk Tracker (OFR-RT) is a React single-page application that provides structured risk and issue tracking for PwC Canada's OFR group. It uses:

  • Azure Static Web Apps for hosting (serverless CDN-based delivery)
  • Microsoft Entra ID for corporate SSO authentication (OAuth 2.0 + PKCE)
  • Microsoft Graph API v1.0 for data operations against SharePoint Online Lists
  • SharePoint Online as the data store (3 custom lists)
  • GitHub Actions for automated CI/CD

Phase 1 Azure & Entra ID Setup (Days 1-2)

Step 1.1 — Create Azure Resource Group

1

Navigate to Azure Portal

Go to portal.azure.com and sign in with your PwC admin account.

2

Create Resource Group

# Using Azure CLI (alternative to portal) az group create \ --name rg-ofr-risk-tracker-prod \ --location canadacentral \ --tags "Application=OFR-Risk-Tracker" "Environment=Production" "CostCenter=OFR"

Step 1.2 — Create Azure Static Web App

3

Create Static Web App Resource

In the Azure Portal:

  1. Search for "Static Web Apps" and click Create
  2. Select subscription and resource group (rg-ofr-risk-tracker-prod)
  3. Name: ofr-risk-tracker
  4. Plan: Free (sufficient for internal tool)
  5. Region: Canada Central (or closest to users)
  6. Source: GitHub (authorize your org)
  7. Organization / Repo / Branch: your GitHub repo and main branch
  8. Build Preset: React
  9. App location: /
  10. Output location: dist
  11. Click Review + Create
Note: Azure will automatically create a GitHub Actions workflow file in your repository. You can customize this later in Phase 4.
4

Record the Deployment Token

After creation, go to the SWA resource and copy the Deployment Token from the Overview blade. Save it as a GitHub repository secret named AZURE_STATIC_WEB_APPS_API_TOKEN.

Step 1.3 — Register Application in Microsoft Entra ID

5

Navigate to Entra Admin Center

Go to entra.microsoft.com → App registrations → New registration

6

Configure App Registration

FieldValue
NameOne Firm Risk Tracker
Supported account typesAccounts in this organizational directory only (Single tenant)
Redirect URI — PlatformSingle-page application (SPA)
Redirect URI — URL (Dev)http://localhost:5173
Redirect URI — URL (Prod)https://ofr-risk-tracker.azurestaticapps.net
7

Configure API Permissions

Go to API permissions → Add a permission:

  1. Microsoft Graph → Delegated permissions
  2. Add: User.Read (usually pre-added)
  3. Add: Sites.ReadWrite.All
  4. Click Grant admin consent for [Tenant] (requires Global Admin or Privileged Role Admin)
8

Record Critical Values

From the app registration's Overview page, record:

# Copy these values for Phase 3 configuration VITE_CLIENT_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx # Application (client) ID VITE_TENANT_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx # Directory (tenant) ID
Important: Do NOT create a client secret. The application uses the SPA/public client flow with PKCE, which does not require a secret. Adding one would be a security concern as it would need to be embedded in client-side code.

Phase 3 Application Configuration & Build (Days 3-4)

Step 3.1 — Clone and Configure the Application

# Clone the repository git clone https://github.com/pwc-canada/ofr-risk-tracker.git cd ofr-risk-tracker # Install dependencies npm ci # Create environment file from template cp .env.example .env

Step 3.2 — Set Environment Variables

Edit the .env file with the values from Phase 1 and Phase 2:

# .env (DO NOT commit this file to git) # Microsoft Entra ID (from Phase 1, Step 8) VITE_CLIENT_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx VITE_TENANT_ID=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx # SharePoint Site Configuration (from Phase 2) VITE_SP_HOSTNAME=pwccanada.sharepoint.com VITE_SP_SITE_PATH=/sites/ofr-risk-tracker
Security Note: While Client ID and Tenant ID are non-secret public identifiers, the .env file should still be in .gitignore to prevent accidental exposure of environment-specific values. Verify .gitignore contains .env.

Step 3.3 — Verify Local Build

# Run the linter npm run lint # Build for production npm run build # Preview the production build locally npm run preview # Or run the dev server npm run dev

Step 3.4 — Test Authentication Locally

  1. Start the dev server: npm run dev
  2. Open http://localhost:5173 in your browser
  3. Click "Sign in with Microsoft"
  4. Authenticate with your PwC account (MFA if required)
  5. Verify you see the authenticated landing with "Connect to SharePoint" button
  6. Click "Connect to SharePoint" to verify data loads from the SP lists
  7. Verify the connection badge shows Connected
Checkpoint: If you see your risk items loaded from SharePoint and can create new intake items, Phase 3 is complete. If SharePoint is empty, that's expected — the app will show an empty state.
Phase 4 CI/CD Pipeline Setup (Days 4-5)

Step 4.1 — Configure GitHub Actions Environment Variables

In your GitHub repository, go to Settings → Secrets and variables → Actions:

Secret / VariableTypeValue

VITE_CLIENT_IDVariableEntra App Client ID
VITE_TENANT_IDVariableEntra Tenant ID
VITE_SP_HOSTNAMEVariableSharePoint hostname

Step 4.2 — GitHub Actions Workflow

Create or update .github/workflows/azure-static-web-apps.yml:

name: Deploy to Azure Static Web Apps on: push: branches: [main] pull_request: types: [opened, synchronize, reopened, closed] branches: [main] env: VITE_CLIENT_ID: ${{ vars.VITE_CLIENT_ID }} VITE_TENANT_ID: ${{ vars.VITE_TENANT_ID }} VITE_SP_HOSTNAME: ${{ vars.VITE_SP_HOSTNAME }} VITE_SP_SITE_PATH: ${{ vars.VITE_SP_SITE_PATH }} jobs: build_and_deploy: if: github.event_name == 'push' || (github.event_name == 'pull_request' && github.event.action != 'closed') runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Setup Node.js uses: actions/setup-node@v4 with: node-version: '20' cache: 'npm' - name: Install and Build run: | npm ci npm run lint npm run build - name: Deploy to Azure SWA uses: Azure/static-web-apps-deploy@v1 with: azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_API_TOKEN }} repo_token: ${{ secrets.GITHUB_TOKEN }} action: "upload" app_location: "/" output_location: "dist" skip_app_build: true close_pull_request: if: github.event_name == 'pull_request' && github.event.action == 'closed' runs-on: ubuntu-latest steps: - name: Close staging environment uses: Azure/static-web-apps-deploy@v1 with: azure_static_web_apps_api_token: ${{ secrets.AZURE_STATIC_WEB_APPS_API_TOKEN }} action: "close"

Step 4.3 — Configure Static Web App Settings

Create staticwebapp.config.json in the project root:

{ "navigationFallback": { "rewrite": "/index.html", "exclude": ["/assets/*"] }, "globalHeaders": { "X-Content-Type-Options": "nosniff", "X-Frame-Options": "DENY", "X-XSS-Protection": "1; mode=block", "Referrer-Policy": "strict-origin-when-cross-origin", "Permissions-Policy": "camera=(), microphone=(), geolocation=()" }, "mimeTypes": { ".json": "application/json" } }

Step 4.4 — Deploy to Staging

# Create a feature branch and push to trigger staging deploy git checkout -b feature/initial-deploy git add . git commit -m "Configure CI/CD pipeline and security headers" git push -u origin feature/initial-deploy # Create a Pull Request via GitHub # Azure SWA will auto-deploy a staging environment
Staging URLs: Azure Static Web Apps automatically creates a preview URL for each pull request, e.g., https://<random>-ofr-risk-tracker.azurestaticapps.net. Add this URL to the Entra ID app's redirect URIs for testing.

8. Rollback Procedures

Application Rollback

Azure Static Web Apps deploys from Git. To rollback:

# Option 1: Revert the problematic commit git revert HEAD git push origin main # GitHub Actions automatically redeploys the reverted state # Option 2: Deploy a specific previous commit git checkout <known-good-commit-hash> # Force push or create a revert PR

RTO: 5-10 minutes (build + deploy time)

SharePoint Data Rollback

  • Individual Items: SharePoint List items have version history; restore from item version history
  • Deleted Items: SharePoint Recycle Bin retains deleted items for 93 days
  • Full List Restore: Contact M365 Admin for site collection restore from Microsoft backup

Entra ID Rollback

  • Disable App: In Entra Admin Center, go to App registrations → select app → Properties → set "Enabled for users to sign-in" to No
  • Revoke Consent: Enterprise applications → select app → Permissions → Revoke admin consent

10. Go-Live Readiness Checklist

Infrastructure

  • Azure resource group created in correct subscription and region
  • Azure Static Web App provisioned and responding
  • Custom domain configured with TLS certificate (if applicable)
  • GitHub Actions deployment token stored as repository secret

Identity & Access

  • Entra ID app registration created (single tenant, SPA platform)
  • Redirect URIs configured for all environments (localhost, staging, production)
  • API permissions granted: User.Read + Sites.ReadWrite.All
  • Admin consent granted by Global Admin or Privileged Role Admin
  • No client secret created (PKCE public client only)

SharePoint

  • SharePoint site created at correct URL path
  • OFR_RiskRegister list created with all columns
  • OFR_UpdateHistory list created with all columns
  • OFR_IntakeQueue list created with all columns
  • ParentItemID column indexed on UpdateHistory list
  • Permission groups configured (Owners, Members, Visitors)
  • Initial users added to appropriate groups

Application

  • Environment variables set in GitHub repository (vars + secrets)
  • Production build succeeds without errors (npm run build)
  • ESLint passes with no errors (npm run lint)
  • staticwebapp.config.json includes security headers
  • .env file in .gitignore

CI/CD

  • GitHub Actions workflow file committed and functional
  • Staging deployment succeeds on pull request
  • Production deployment succeeds on merge to main
  • Dependabot configured for npm dependency alerts

Testing

  • All test cases (T-001 through T-016) passed
  • UAT completed by OFR Risk Committee members
  • Critical UAT feedback addressed
  • Security headers verified in browser DevTools

Documentation & Compliance

  • Security Design Document (SDD) completed and submitted
  • Application Readiness Review (ARR) initiated
  • Go-live communication drafted and approved
  • Support channel established (Teams / email)
  • Rollback procedures documented and tested
