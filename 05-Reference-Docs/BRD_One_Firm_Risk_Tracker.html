1. Executive Summary

The One Firm Risk Management Office currently manages cross-firm risk issues through a manually maintained Word document that combines ad-hoc "issues of the day" bullet points with a tracking table. This process suffers from three systemic failures: issues enter the document but never leave it, updates are inconsistent and unverifiable, and there is no passive mechanism to identify stale or neglected items.

This BRD defines the requirements for a web-based Risk & Issue Tracker that replaces the Word document with a structured intake-triage-track-archive workflow. A functional React/Vite prototype has been built and validated with sample data. This document specifies the requirements for moving from prototype to production within the existing Microsoft 365 environment.

Key Outcome: A bilingual (EN/FR), SharePoint-integrated risk tracking tool that creates passive accountability through automated staleness indicators, streamlines weekly meeting preparation, and provides an auditable history of every risk item from intake through closure and archival.

2.2 Desired Future State

A single web application accessible through SharePoint Online that provides:

  1. Structured intake — New issues enter through a triage panel with required fields (title, owner, priority, description) before being promoted to the formal register
  2. Tracked lifecycle — Every item moves through defined statuses (New → Active → Monitoring/Escalated → Closed) with timestamped update history
  3. Passive accountability — Automatic "Days Since Update" calculation with red/amber/green staleness indicators that make neglected items immediately visible
  4. Meeting-ready dashboard — Summary view showing open item counts, stale items, and priority/status breakdowns for weekly review preparation
  5. Persistent storage — All data backed by SharePoint Lists or Dataverse with automatic archival of closed items
  6. Bilingual UI — Full English/French interface toggle built into the design system

6.2 Auto-Archive Strategy

When an item's status is changed to "Closed":

  1. Item remains visible in the register for 30 days with "Closed" status badge
  2. After 30 days, a scheduled Power Automate flow moves the item to OFR_RiskRegister_Archive (identical schema)
  3. Associated update history entries are copied to OFR_UpdateHistory_Archive
  4. Archived items are accessible via a separate "Archive" view but do not appear in active dashboard counts

8. Assumptions & Constraints

8.1 Assumptions

  • PwC Canada M365 tenant supports SharePoint Framework (SPFx) deployment or Azure Static Web Apps with MSAL
  • An Azure App Registration can be created or an existing one can be used for MSAL authentication
  • The target SharePoint site collection has sufficient storage and list threshold capacity
  • Power Automate is available for scheduled archive flows
  • Users have modern browsers (Edge or Chrome) on corporate devices

8.2 Constraints

  • SharePoint List item limit of 5,000 items per view (mitigated by indexed columns and archive strategy)
  • No external hosting infrastructure — must run entirely within M365 ecosystem
  • Bilingual requirement adds approximately 30% to UI development effort
  • Phase 1 budget limited to internal team capacity — no external vendor engagement
