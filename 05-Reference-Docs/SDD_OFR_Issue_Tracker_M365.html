0. About This Document

0.1 Purpose

This Security Design Document (SDD) describes the security architecture, controls, and operational requirements for the OFR Issue Tracker application. The OFR Issue Tracker is a fully M365-native solution built with Power Apps, Power Automate, and SharePoint Online that provides structured risk and issue tracking for PwC Canada's One Firm Risk group, replacing both a manually maintained Word-based register and the earlier React/Vite SPA prototype.

The M365-native architecture eliminates all custom code, Azure subscriptions, CI/CD pipelines, and external infrastructure. The entire solution operates within the existing Microsoft 365 Business Standard subscription using Power Apps Developer and Power Automate Free licensing.

This document is authored in accordance with the PwC Security Design Document for Applications v1.5 template and is intended to support the Application Readiness Review (ARR) process, including completion of the Application Compliance Questionnaire (ACQ).

1. Solution Description

1.1 Application Description

The OFR Issue Tracker is a fully M365-native application built on the Power Platform that provides:

  • Risk Intake Queue — structured submission and triage of new risk issues with promote/dismiss workflow
  • Issue Lifecycle Tracker — filterable, sortable, searchable tracker with staleness indicators (color-coded days since last update)
  • Update Workflow — status updates with full audit history timeline on each issue
  • Dashboard — KPI summary cards for Open Items, Stale issues, and priority breakdown (High/Medium/Low)
  • Automated Staleness Calculation — daily scheduled flow computes days since last update and sets staleness flags
  • Automated Intake Promotion — instant flow creates tracker entries from intake items with audit trail

The application uses Microsoft Entra ID for automatic corporate SSO authentication (built into the Power Platform), SharePoint Online Lists as the data store, Power Apps Canvas App as the user interface, and Power Automate Cloud Flows for automation.

Architecture Advantage: Zero custom code. Zero Azure subscriptions. Zero CI/CD pipelines. Zero custom domains. Zero SSL certificates. Everything runs within M365. Users authenticate automatically via Entra ID. Data stays in SharePoint. The app lives in Power Apps and can be pinned to Teams. IT has nothing new to manage.

3. Application Security Controls

3.1 Authentication & Access Methods

The application uses Microsoft Entra ID as the sole identity provider. Authentication is handled automatically by the Power Platform — there is no custom authentication code, no MSAL library, no token management, and no client-side credential handling.

Authentication Configuration

  • Protocol: OAuth 2.0 / OpenID Connect (managed by Power Platform)
  • Flow: Automatic SSO via Power Apps service; user logs in once to M365
  • Library: None — authentication is platform-managed (no MSAL, no custom code)
  • Authority: M365 tenant Entra ID (automatic)
  • Token Cache: Managed by Power Platform service; no tokens in browser storage
  • Cookie Usage: Standard M365 session cookies (platform-managed)
  • MFA: Enforced via Entra ID Conditional Access policies (tenant-level)
  • Custom App Registration: Not required — Power Apps uses first-party Microsoft app registration

Authentication Methods by Account Type

Key Design Decision: The M365-native architecture eliminates all custom authentication code. There is no Entra ID app registration to manage, no client secrets, no redirect URIs, and no MSAL configuration. Authentication is fully delegated to the Power Platform, which uses Microsoft's first-party app registrations internally. Power Automate flow connections use the flow creator's delegated credentials via the SharePoint connector.

3.7 Security Stack

1. Access to Application

  • Microsoft Entra ID: Sole identity provider; automatic SSO via Power Platform
  • Conditional Access: Tenant-level MFA, device compliance, location restrictions
  • Power Apps Sharing: App only accessible to explicitly shared users/groups
  • SharePoint Permissions: Site membership controls data layer access

2. Network Security

  • HTTPS-Only: All Microsoft services enforce HTTPS
  • TLS 1.2+: Minimum version enforced by all Microsoft endpoints
  • M365 Tenant Isolation: All data and services within single tenant boundary
  • No External APIs: No cross-origin requests; no third-party service integrations

3. Data Security

  • Encryption at Rest: AES-256 via Microsoft platform-managed keys
  • Encryption in Transit: TLS 1.2+ for all communications
  • No Client-Side Storage: No tokens, credentials, or data cached in browser storage by the application
  • Data Residency: SharePoint data stored in Canadian datacenter region per M365 tenant configuration

4. Application Security

  • No Custom Code: Entire application built with Power Platform low-code tools; no JavaScript, no HTML injection vectors
  • Input Validation: Power Apps form controls enforce data types, required fields, and choice constraints
  • XSS Prevention: Power Apps rendering engine auto-sanitizes all displayed content
  • Connector DLP: Power Platform DLP policies restrict available connectors to approved list

5. Container / Cluster Security

Not Applicable: The OFR Issue Tracker is deployed entirely within Microsoft's managed Power Platform and SharePoint Online services. There are no containers, Kubernetes clusters, VMs, or server-side compute resources. This section is included for SDD template completeness.
