1. Purpose

A working prototype of the One Firm Risk Tracker has been built and validated using React, Vite, and Tailwind CSS. It demonstrates the full issue lifecycle workflow: intake triage, active tracking with staleness indicators, update history, dashboard metrics, bilingual UI, and CSV export.

This document presents four deployment options for moving the prototype to production within PwC Canada's Microsoft 365 environment. Each option is evaluated on speed to deploy, integration depth, infrastructure requirements, licensing implications, and long-term scalability.

Recommendation: Option A (Azure Static Web App + MSAL + Microsoft Graph) is the recommended short-term path. It preserves the existing codebase, deploys in 2–3 weeks, requires no infrastructure beyond a free-tier Azure resource, and connects directly to SharePoint Lists for persistent storage. Option B (SPFx) should be considered for Phase 2 if deeper SharePoint/Teams embedding is required.

Regardless of deployment option, the production architecture follows the same logical pattern:

The key decision is how and where the React application is hosted, which determines the authentication method, build toolchain, and integration depth with SharePoint and Teams.

4. Deployment Options

Option A: Azure Static Web App + MSAL + Graph API
Standalone SPA — Keep Existing React/Vite Stack
Recommended

How It Works

The existing React/Vite application is deployed as a static web app on Azure. Users navigate to a URL (e.g., https://ofr-tracker.azurestaticapps.net) and authenticate with their PwC M365 credentials via MSAL.js. The app calls Microsoft Graph API to read/write SharePoint Lists for data persistence.

What Changes in the Prototype

  1. Add @azure/msal-browser and @azure/msal-react packages (~2 days)
  2. Wrap App component in MsalProvider with Entra ID app registration (~1 day)
  3. Replace in-memory state with Graph API calls to SharePoint Lists (~5–7 days)
  4. Configure Azure Static Web App resource and GitHub Actions CI/CD (~1 day)
  5. Create three SharePoint Lists matching the data model in the BRD (~1 day)
  6. Configure Power Automate archive flow for closed items (~1 day)

Key Advantages

  • Preserves entire existing codebase — Vite, Tailwind, React components transfer as-is
  • Free-tier hosting — Azure Static Web Apps free plan includes custom domains, SSL, and global CDN
  • No webpack migration — continues using Vite's fast build and HMR
  • Fastest time to production — estimated 2–3 weeks from today
  • Full Graph API access — can expand to Teams, Planner, or other M365 services

Limitations

  • App lives at a separate URL, not embedded inside SharePoint pages (can be bookmarked or linked from a SharePoint page)
  • Requires an Azure App Registration in Entra ID (one-time admin setup)
  • Requires an Azure subscription (free tier is sufficient for this use case)
Browser --[MSAL SSO]--> Entra ID | | Bearer Token v Azure Static Web App SharePoint Online (React + Vite build) --[Graph API]--> Lists (CRUD) Free tier, global CDN Power Automate (archive)
Option B: SharePoint Framework (SPFx) Web Part
Native SharePoint Integration — Requires Toolchain Migration
Best Integration

How It Works

The React components are ported into a SharePoint Framework (SPFx) project, which uses Heft + webpack (as of SPFx v1.22) instead of Vite. The web part is deployed to the SharePoint App Catalog and can be added to any SharePoint page, Microsoft Teams tab, or Viva Connections dashboard. Authentication is automatic (SSO) — no MSAL setup required.

What Changes in the Prototype

  1. Scaffold a new SPFx v1.22 project using yo @microsoft/sharepoint (~1 day)
  2. Port React components from the Vite project into the SPFx web part render tree (~3–5 days)
  3. Replace Tailwind CSS with SPFx-compatible styling (CSS Modules or Fluent UI) (~3–5 days)
  4. Use SPFx's built-in SPHttpClient for SharePoint REST calls (~3–5 days)
  5. Package, deploy to App Catalog, and sync to Teams (~1 day)

Key Advantages

  • Deepest SharePoint integration — web part runs natively inside SharePoint pages
  • Automatic SSO — no MSAL configuration, no app registration for auth (SPFx handles it)
  • Cross-platform deployment — same package works in SharePoint, Teams tabs, and Viva
  • No separate hosting — assets are served from the SharePoint CDN automatically
  • IT-admin governed — deployed through the App Catalog, centrally managed

Limitations

  • Cannot use Vite — SPFx mandates Heft/webpack; the build toolchain must change
  • Tailwind CSS migration — SPFx's CSS Modules approach requires reworking the styling layer
  • Longer timeline — estimated 4–6 weeks due to toolchain migration and styling rework
  • SPFx version dependency — must track Microsoft's SPFx release cadence for Node.js and React compatibility
  • Tenant admin approval — SPFx deployment to the App Catalog requires admin consent
SharePoint Page / Teams Tab / Viva | | SPFx runtime (automatic SSO) v SPFx Web Part SharePoint Online (Heft + webpack build) --[SPHttpClient]--> Lists (CRUD) Hosted on SP CDN Power Automate (archive)
Option C: Power Platform with Dataverse Backend
Low-Code / Pro-Code Hybrid — Maximum Data Capability
Most Scalable

How It Works

The data layer moves from SharePoint Lists to Dataverse, which provides relational data, role-based security, business rules, and native integration with Power Automate, Power BI, and Copilot. The front-end can be delivered as either: (a) a custom React app calling the Dataverse Web API, or (b) a Power Apps model-driven app with Power Apps Component Framework (PCF) controls for the custom visualization (dashboard, staleness indicators).

What Changes in the Prototype

  1. Design Dataverse tables (RiskItem, UpdateEntry, IntakeItem) with relationships (~2 days)
  2. Option C1: Keep React app, replace Graph calls with Dataverse Web API (OData) (~5–7 days)
  3. Option C2: Build a model-driven Power App and create PCF controls for custom UI elements (~8–12 days)
  4. Configure Dataverse security roles for read/write access control (~2 days)
  5. Build Power Automate flows for archive and (optionally) notifications (~2 days)

Key Advantages

  • Relational data model — proper foreign keys, cascading deletes, calculated columns
  • Built-in role-based security — row-level and column-level access control without custom code
  • Business rules engine — validation, auto-populate, and workflow logic without custom code
  • Native Power BI integration — connect Power BI directly to Dataverse for advanced reporting
  • Citizen developer expansion — non-developers can extend the app via Power Apps and Power Automate
  • AI-ready — Copilot integrations available natively in Dataverse/Power Platform

Limitations

  • Licensing cost — Dataverse access requires Power Apps Per User ($20/user/month) or Per App ($5/app/user/month) licenses unless covered by existing D365/M365 entitlements
  • Longer timeline — estimated 6–8 weeks for the full Dataverse + custom UI implementation
  • Higher complexity — Dataverse administration, solution management, and environment strategy add operational overhead
  • PCF limitations — PCF components run inside Power Apps contexts, not as standalone web apps
React SPA (Option C1) OR Power App (Option C2) | | | MSAL + OData | Native auth v v Dataverse +-- RiskItem (table, relational) +-- UpdateEntry (child table) +-- IntakeItem (table) +-- Security Roles, Business Rules +-- Power Automate (archive + notifications) +-- Power BI (reporting)
Option D: Hybrid — Standalone SPA Embedded in SharePoint
Quick Win + SharePoint Presence — Iframe Approach
Compromise

How It Works

Combines Options A and B: the React/Vite app is deployed as an Azure Static Web App (Option A), but is embedded inside a SharePoint page using the out-of-the-box Embed web part (iframe). Users access the app from within their existing SharePoint site, while the app itself runs in its own hosting environment.

Key Advantages

  • Best of both worlds — keep Vite/React stack while appearing inside SharePoint
  • Faster than Option B — no SPFx toolchain migration needed
  • Familiar location — users find the tracker on a SharePoint page they already visit

Limitations

  • Iframe isolation — no seamless SSO pass-through; users may see a second auth prompt
  • Limited SharePoint context — the embedded app cannot access SharePoint page context or theme
  • Not officially recommended — Microsoft recommends native SPFx for embedded SharePoint experiences
  • Visual seam — iframe borders and scroll bars can feel disconnected from the host page

6. Recommended Approach: Phased Deployment

Phase 1: Immediate (Weeks 1–3) — Option A

Deploy the existing React/Vite application as an Azure Static Web App with MSAL authentication and Microsoft Graph API connected to SharePoint Lists. This delivers a working, authenticated, data-persistent tool with zero build toolchain changes.

Week 1
Infrastructure Setup: Create Entra ID app registration with Sites.ReadWrite.All scope. Provision Azure Static Web App (free tier). Create three SharePoint Lists (RiskRegister, UpdateHistory, IntakeQueue) on the target site. Set up GitHub Actions for CI/CD.
Week 2
Integration Build: Add MSAL React wrapper to the application. Build a data service layer that replaces in-memory state with Graph API CRUD operations against the SharePoint Lists. Handle token acquisition, error states, and loading indicators.
Week 3
Testing & Launch: End-to-end testing with real SharePoint data. Configure Power Automate archive flow. User acceptance testing with the Risk Management team. Go-live with the production URL shared via Teams/email.

Phase 2: Enhancement (Weeks 4–8) — Evaluate Option B or Stay

Based on Phase 1 user feedback, decide whether the standalone URL is sufficient or if native SharePoint/Teams embedding is required. If embedding is needed, begin the SPFx port (Option B). Otherwise, enhance the Phase 1 deployment with:

  • Power Automate notifications for stale items (email + Teams adaptive card)
  • Teams tab registration for the Azure Static Web App URL
  • "My Items" filter using the authenticated user's identity
  • Mobile-responsive layout adjustments

Phase 3: Scale (Months 3–6) — Evaluate Option C

If the tool is adopted across additional risk domains, or if relational data integrity, role-based access control, or Power BI reporting become requirements, evaluate migrating the data layer from SharePoint Lists to Dataverse (Option C). The React front-end can remain; only the API layer changes from Graph to Dataverse Web API.

9. Technical Appendix: Key Integration Details

9.1 MSAL React Setup (Option A)

The Microsoft Authentication Library for React provides a provider pattern that wraps the application and handles token lifecycle:

  • Install: @azure/msal-browser + @azure/msal-react
  • Configure with Client ID and Tenant ID from Entra ID app registration
  • MsalProvider wraps the App component; AuthenticatedTemplate / UnauthenticatedTemplate handle conditional rendering
  • Token acquisition via acquireTokenSilent() with fallback to acquireTokenRedirect()
  • Redirect URI set to the Azure Static Web App URL

9.2 Microsoft Graph API Patterns (Option A)

SharePoint Lists are accessed through Graph endpoints:

  • GET /sites/{site}/lists/{list}/items — read all items
  • POST /sites/{site}/lists/{list}/items — create a new item
  • PATCH /sites/{site}/lists/{list}/items/{id} — update an item
  • Sites can be addressed by URL: /sites/pwc.sharepoint.com:/sites/ofr-risk
  • Permissions required: Sites.ReadWrite.All (delegated) or the more granular Sites.Selected

9.3 SPFx Toolchain (Option B)

As of SPFx v1.22 (current), the build system has migrated from Gulp to Heft:

  • Scaffolding: yo @microsoft/sharepoint
  • Build: heft build --production (replaces gulp bundle)
  • Package: heft package-solution --production
  • Requires Node.js 22 LTS, webpack 5 (cannot be replaced with Vite)
  • Deployment via .sppkg package uploaded to the App Catalog
  • Sync to Teams: one-click from the App Catalog ribbon

9.4 Dataverse Web API (Option C)

  • OData v4 RESTful API: https://{org}.api.crm.dynamics.com/api/data/v9.2/
  • Authentication: same MSAL flow as Graph, different scope (https://{org}.crm.dynamics.com/.default)
  • Full CRUD, filtering, sorting, expanding related entities
  • Batch operations supported for multi-item updates
  • React SPAs use the same @azure/msal-browser package

9.5 Power Automate Archive Flow

A scheduled flow (daily or weekly) that:

  1. Queries OFR_RiskRegister for items where Status = "Closed" and LastUpdated < 30 days ago
  2. Copies each qualifying item to OFR_RiskRegister_Archive
  3. Copies associated OFR_UpdateHistory records to OFR_UpdateHistory_Archive
  4. Deletes the originals from the active lists
  5. Sends a summary notification to the Risk Management Teams channel
